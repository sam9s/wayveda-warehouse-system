WayVeda / Shiprocket Review

Shiprocket Mapping Gap Analysis

This document explains the current Shiprocket integration result and the specific product-name ambiguities that still need business confirmation before the remaining dispatch rows can be imported safely.

Current Live Result

  • Shiprocket authentication is working.
  • Dispatch sync is working.
  • 197 Shiprocket orders created 210 dispatch rows in the WayVeda system.
  • 60 orders are still being held back because their product names are not safe to map automatically.
These held-back rows were not lost. They were deliberately skipped so inventory is not deducted from the wrong product.

What Is Working

  • Shiprocket orders can now be read from the live API.
  • Dispatch-stage orders are reaching the WayVeda system.
  • Mapped products are already creating real dispatch movements.
  • Those imported dispatches are now affecting ledger and dispatch analysis.

Why We Are Not Guessing

  • If we map a Shiprocket product to the wrong SKU, inventory will reduce from the wrong product.
  • That would corrupt stock balances and dispatch reporting.
  • So the integration follows a safe rule:
    • clear match = import
    • unclear match = hold back for confirmation

What We Need From WayVeda

  1. Confirm what Male Enhancer maps to in the canonical catalog.
  2. Confirm whether the combo name is a multi-product bundle.
  3. If it is a bundle, provide the exact product breakdown and quantity mapping.
  4. Confirm whether any additional Shiprocket marketing names are expected for the same SKUs.

What Happens After Confirmation

  1. The approved mappings will be added to the Shiprocket mapping file.
  2. Shiprocket sync will be run again.
  3. Previously held-back orders will then import into the dispatch ledger.
  4. Duplicate-safe sync logic will prevent already imported orders from being inserted again.

Decision Rule

Until WayVeda confirms the ambiguous names, those rows should remain blocked. That is safer than guessing and damaging inventory accuracy.
